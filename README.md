# Тестовое задание
## Что реализовано

1. Регистрация пользователя
2. Логин / логаут
3. Идентификация пользователя по токену
4. Редактирование профиля
5. Мягкое удаление аккаунта (`is_active = 0` + инвалидация сессий)
6. система ролей и прав доступа
7. Mock-бизнес-ресурсы с проверкой прав
8. Admin API для просмотра и изменения правил доступа


Swagger:
- `http://127.0.0.1:8000/docs`

## Тестовые пользователи

- Администратор:
  - email: `admin@example.com`
  - password: `Admin123!`
- Обычный пользователь:
  - email: `user@example.com`
  - password: `User123!`

## Схема БД и логика разграничения доступа

### Таблица `users`
- Хранит профиль пользователя и флаг активности `is_active`.
- При «удалении» пользователь не удаляется физически, а деактивируется.

### Таблица `sessions`
- Хранит токены авторизации после логина.
- По токену система определяет, кто выполняет запрос.
- При logout или soft delete сессии инвалидируются (`is_active=0`).

### Таблица `roles`
- Роли (`admin`, `user` и т.д.).

### Таблица `permissions`
- Право = пара `resource + action`.
- Пример: `projects:read`, `rules:write`.

### Таблица `role_permissions`
- Связь многие-ко-многим между ролями и правами.

### Таблица `user_roles`
- Связь многие-ко-многим между пользователями и ролями.


## Основные API

### Auth / User
- `POST /auth/register` — регистрация
- `POST /auth/login` — вход
- `POST /auth/logout` — выход
- `GET /users/me` — получить профиль
- `PATCH /users/me` — обновить профиль
- `DELETE /users/me` — мягко удалить аккаунт

### Access rules (для администратора)
- `GET /admin/roles` — список ролей
- `GET /admin/permissions` — список прав
- `GET /admin/rules` — карта ролей/прав и пользователей/ролей
- `POST /admin/roles/{role_id}/permissions` — выдать роли право
- `DELETE /admin/roles/{role_id}/permissions/{permission_id}` — забрать право у роли
- `POST /admin/users/{user_id}/roles` — выдать пользователю роль
- `DELETE /admin/users/{user_id}/roles/{role_id}` — забрать роль у пользователя
